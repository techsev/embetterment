What I wanted to do was have a way to quickly and dynamicly have progress bars tracking my skills and goals that could easily be modified by changing 1 - 2 variables. I considered making it database driven or using PHP however, my goal for this site is to try techniques i’ve never used before, including making an entire static website.

So instead I decided to use SASS and Compass. That way I can make changes fairly quickly on, compile the CSS and then push to git pages.

Where the CSS differs for each progress bar is the width, color and the percentage amount.

I created a SASS mixin that when given a percentage, would set it as the inner-div’s width, the “content” field of the div:after class as well as modify the background color of the inner div between the colors red and green.

Then all I have to do was supply the required percentage for each bar, which in some cases I just supplied directly or in others, I used SASS variables and math to generate the percentage. I had a little fun with my SASS Progress Bar Variables

▼ Static Page Templating // 11/11/13

Now that I have a few more projects to display on my site, it was getting difficult to keep track of everything in one single file. I could have created a templating system in either PHP or Javascript, however, I've been hearing a lot about blog aware static page generators that give you the benefit of a static pages without having to hand code each file.

After looking over how Jekyll laid out it's structure, I decided to build my own grunt script using similar concepts. I currently have it working for projects but I also plan to use it for my eventual blog as well.

Static Page Templating

The first thing I needed to do was create a few new directories:

  • _projects: Where the project snippets go
  • _posts: Where the (eventual) blog posts will go
  • _templates: Where the main index.html template go

The template file contains templating variables, at the moment, only {{PROJECTS}}. This is where the projects .html files contained in "_PROJECTS" will be inserted

Next I created a grunt task that reads in all the files in the "_PROJECTS" folder. Using the concepts used in Jykyll, I named all my project files starting with the date they were created followed by tge project name. For example this file is named "11-11-2013-static-page-templating.html" What this lets me do is sort the file in order of date from newest to oldest.

With all the project .html files read into a single variable, I'm able to do a simple search and replace using the primary template file then write this new file to index.html in the primary directory.

At any point if I want to add a new project snippet, I just need to save it to the "_PROJECTS" directory and run grunt and it will be added in order with the rest of the projects.

The steps are essentially the same for the blog however, I haven't yet figured out how I want to handle my blog. That's a project for another day.

This is a test of a different way to handle Responsive Web Design using Javascript and Cascading Style Sheets. My idea is to make the overall text smaller based on media queries.

The words highlighted in green are the full length words, if the words are highlighted in red, then you're seeing the shortend words.

CSS Responsive Words

This proof of concept script is just something I threw together on my lunch break at work. The idea is to abbreviate words when the screen size of the device is small. Currently just a proof of concept and should not be used on a live site.

Should this script run on the client side? Probably not, I'll probably end up making this into a grunt task if I ever find it to be a good idea.

Should this concept be used in real world websites? Probably not, since any search engine crawlers coming to the site might get different version of the content (It's rumored that google crawlers can now handle javascript and media queries)

I'm considering changing the way I place the alternate text into the html. Currently I wrap the original word in a span, and also append a new span with alt word, but now that means every replaced word is in the code twice. The alternative would be to have the javascript generate CSS styles with span:after tags that in the content field contain the shortened word, but then I lose browser support for IE. For now it works.

▼ Live Data Progress Bar // 10/10/13

Dynamic Progress Bars using live data
Oct 10th, 2013

Now that I’ve created the progress tracker, I wanted to be able to update it in a few different ways.

  • I want to be able to just change a variable at any time manually.
  • I want to be able to add an project entry to my site that will increase my progress by adding the original value plus my gained “experience”, like a role playing game.
  • I want to be able have certain progress bars be updated by referencing certain APIs and other resources.

Option 1 was doable but not to my liking, rather not have to open a scss file for each change, but for now it’s manageable.

Option 2 will be trickier, since I’m not planning to have the site use any back end database. I will get to this one later.

So I’ve decided to work on option 3 for now. I’m going to use node.js and grunt to pull data from twitter to update my weight progress.

Ideally I wanted to have grunt pull the data from the web service “My Fitness Pal” since I currently use it to track my weight progress, however, their API tools are currently private. However, they do allow for automatic posting to twitter, so I created a twitter account, “SeveIsFat” that was exclusively for this purpose. From there I created a node.js script that pulls the most recent tweet on my account, strips the everything but the amount of weight I’ve lost since I started tracking, and updates a file that holds exclusively that variable which then gets include imported into my _progress.scss file, which is where all my current progress goal variables are saved.

Once / If I get the My fitness Pal API keys I’ll update this to allow for a direct link to the data, but for now it’s the best I could think of.
